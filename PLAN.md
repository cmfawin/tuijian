# 推荐看板口径重构方案

## Summary
- 将当前单一“推荐加权 UV价值 / 转化率”拆为两个算法池：
  - `全链路算法盘`
  - `排序算法盘`
- 不再引入 `core cohort` 概念；模块是否纳入口径仅由 `模块类型 + algorithmStartMonth` 决定。
- `全链路算法盘` 继续用加权指标做主分析口径。
- `排序算法盘` 的 UV 变化主表达改为 `堆叠贡献柱 + 总线`，直接展示各模块对盘面变化的贡献来源。
- 历史月份中，尚未接算法的模块直接隐藏，不进入卡片、趋势、矩阵、总览表和归因。

## Key Changes
### 1. 数据口径
- 新增模块元数据配置，来源于 `模块类型表.xlsx`，至少包含：
  - `模块名`
  - `类型`：`全链路算法` / `排序算法`
  - `algorithmStartMonth`
- 月度取数时，模块仅在 `月份 >= algorithmStartMonth` 时才视为“已生效”。
- 继续排除 `全站`、`搜索结果`、`推荐加权` 原始行，不把它们当作推荐模块参与池内计算。

### 2. 指标计算
- `全链路算法盘加权UV价值`：
  - 仅对当月已生效的全链路模块，按池内 `引导成交业绩占比` 加权 `UV价值`
- `全链路算法盘加权支付转化率`：
  - 仅对当月已生效的全链路模块，按池内 `引导成交业绩占比` 加权 `支付转化率`
- `排序算法盘加权UV价值 / 加权支付转化率`：
  - 同样仅对当月已生效的排序模块，在排序池内独立加权
- 排序盘堆叠图的每个模块柱段值定义为：
  - `模块贡献 = 模块在排序池内的业绩权重 × 模块UV价值`
  - 所有模块贡献求和后，等于当月 `排序算法盘加权UV价值`
- `常买的推荐` 直接从 `202603` 纳入全链路池，不做特殊对冲处理。

### 3. 页面结构
- 顶部改为 `5` 张核心卡：
  - `全链路加权UV价值`
  - `全链路加权支付转化率`
  - `排序盘加权UV价值`
  - `排序盘加权支付转化率`
  - `全站UV价值`
- 不再单独保留“模块数”卡；在全链路 / 排序盘卡片说明中展示当前月有效模块数，例如 `18个`、`5个`
- 趋势区支持上述 `5` 个指标切换查看
- 归因区仅保留 `全链路算法盘 UV价值月环比归因`
- `排序算法盘` 新增独立分析区：
  - 主图为 `堆叠贡献柱 + 总加权UV价值线`
  - 每个模块固定颜色
  - 在 `202512 / 202601 / 202603` 等模块纳入月份增加口径事件标注
- 模块总览表保留一张，但仅显示当前月已生效模块，并增加模块类型标识

### 4. 现有模块归属
- 按 `模块类型表.xlsx` 执行当前分组：
  - `排序算法` 共 `5` 个：
    `凑单浮层`、`分类页主商品`、`券适用商品列表页`、`爆款专区落地页`、`首页爆款专区`
  - 其余为 `全链路算法`
- 新纳入排序池的模块起算月：
  - `分类页主商品`：`202512`
  - `券适用商品列表页`：`202601`
  - `凑单浮层`：`202603`

## Public Interfaces / Types
- 新增前端模块配置结构，例如：
  - `moduleMetaMap[moduleName] = { type, algorithmStartMonth }`
- 新增按池过滤和按月生效判断的计算入口，例如：
  - `getActiveModulesByPool(month, poolType)`
  - `calcWeightedByPool(month, poolType)`
  - `calcSortingContributionSeries()`
- 现有使用“推荐模块全集”的计算逻辑，需要统一改为“按池 + 按起算月”过滤后再计算。

## Test Plan
- 数据校验
  - 校验 `202511` 时 `分类页主商品 / 券适用商品列表页 / 凑单浮层 / 常买的推荐` 均不进入任何算法池
  - 校验 `202512` 时 `分类页主商品` 进入排序池，`券适用商品列表页 / 凑单浮层 / 常买的推荐` 仍未进入
  - 校验 `202603` 时四个新增模块均按配置进入对应池
- 指标校验
  - 抽查 `202603` 的全链路池、排序池加权UV价值与加权支付转化率，确认分池后分母均为池内 GMV 总和
  - 校验排序盘堆叠贡献柱各模块贡献求和，等于排序盘总加权UV价值
- 交互校验
  - 切换到历史月份时，未接算法模块不在卡片统计、矩阵、表格、详情中出现
  - 排序盘图中模块颜色跨月份保持一致
  - `202512 / 202601 / 202603` 的纳入口径标注显示正确
- 展示校验
  - 顶部 `5` 卡在一行内显示稳定
  - 全链路归因区仍能正确输出拖累项/拉升项
  - 排序盘分析区能清楚读出是哪个模块带来的变化

## Assumptions
- `模块类型表.xlsx` 是本次分池与起算月的唯一准入配置源。
- 不引入 `compareStartMonth`、`core cohort` 或固定可比池。
- 排序盘的“变化解释”以贡献堆叠图为主，不再额外复制一整套归因大表。
- 当前版本重点重构 `UV价值` 的表达；排序盘 `支付转化率` 先保留为指标卡和趋势口径，不额外做堆叠分解图。
